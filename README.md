# Cypress Automation – Demoblaze

This project contains automated end to end tests for [demoblaze.com](https://www.demoblaze.com), an online electronics store. The tests cover the most critical user flows for signing up, logging in, and purchasing a laptop. Built from scratch using Cypress with the Page Object Model pattern.

**Author:** Shubham Pandya

---

## Why Cypress

Cypress was chosen over other automation tools for a few practical reasons:

- **JavaScript native** — no language context switching; tests live alongside the app in the same ecosystem
- **Real browser execution** — tests run inside the browser, not through WebDriver, which means more accurate simulation of real user behaviour
- **Automatic waiting** — Cypress retries commands and assertions until they pass or time out, eliminating the need for manual waits in most cases
- **Time-travel debugging** — the Cypress GUI shows a snapshot of the DOM at every command step, making failures fast to diagnose
- **Built-in network control** — `cy.intercept()` makes it straightforward to wait for API calls and assert on network activity without third-party tools
- **Low setup overhead** — a working test suite can be up and running with a single `npm install`

## Running the Tests

The easiest way to run the tests, especially if it is your first time, is through the Cypress GUI:

```bash
npm run cy:open
```

A window will pop up  Click E2E Testing, pick Chrome as your browser, and you will see the test files listed. Click any spec to watch it run.

To run everything headlessly from the terminal:

```bash
npm run cy:run
```

To run a specific file:

```bash
npx cypress run --spec "cypress/e2e/auth/login.cy.js"
npx cypress run --spec "cypress/e2e/auth/signup.cy.js"
npx cypress run --spec "cypress/e2e/purchase/purchaseLaptop.cy.js"
npx cypress run --spec "cypress/e2e/flows/fullUserJourney.cy.js"
```

## Reports and Screenshots

After running `npx cypress run`, the following are generated automatically:

**Test Report**
- Located at `cypress/reports/index.html`
- Open it in any browser to view pass/fail results with charts and embedded screenshots
- Generated by Mochawesome

**Screenshots**
- Located at `cypress/screenshots/`
- Captured after every test and organised into `passed/` and `failed/` subfolders
- Screenshots are also embedded directly in the HTML report

## Configuration

Tests are configured to automatically retry on failure — up to 2 times in `run` mode and 1 time in `open` mode. This reduces flakiness caused by network delays or timing issues on demoblaze's side without masking real failures.

## CI/CD Setup (GitHub Actions)

To run your tests automatically on every push, create the following file in your project:

**.github/workflows/cypress.yml**

```yaml
name: Cypress E2E Tests

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  cypress-run:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 18
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run Cypress tests
        run: npx cypress run

      - name: Upload test report
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: mochawesome-report
          path: cypress/reports/

      - name: Upload screenshots
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: cypress-screenshots
          path: cypress/screenshots/
```

Once this file is pushed to GitHub, the pipeline will trigger automatically on every push to `main`. To view results:

1. Go to your repository on GitHub
2. Click the **Actions** tab
3. Select the latest workflow run
4. Download **mochawesome-report** or **cypress-screenshots** from the Artifacts section

## End-to-End Flow

The full user journey test (`fullUserJourney.cy.js`) covers the complete critical path of the application in a single test:

1. **Sign up** — registers a new unique user using `Date.now()` to avoid username collisions
2. **Assert signup alert** — verifies `"Sign up successful."` appears
3. **Log in** — authenticates with the newly created credentials
4. **Verify logged in state** — checks the welcome message contains the username
5. **Navigate to Laptops** — clicks the Laptops category and waits for the `POST /bycat` API response before proceeding
6. **Select first product** — dynamically picks the first listed laptop
7. **Verify product details** — asserts product name and price on the product page
8. **Add to cart** — clicks Add to cart, waits for `POST /addtocart` network call, and asserts `"Product added."` alert
9. **Go to cart** — navigates to the cart page
10. **Place order** — fills in the order form with fixture data
11. **Verify order confirmation** — asserts the confirmation popup contains the correct name, card, and purchase amount

## My Approach

### Why these flows

The challenge specifically asked for login and purchase flows, so that is where I started. Within those two flows I wanted to make sure I was not just covering the happy path. Empty fields, wrong credentials, missing required information these are the scenarios real users actually run into, and they are the ones most likely to slip through if nobody is considering all of them.

### How I structured the tests

I used the Page Object Model pattern. Every page has its own file under `cypress/pages/` that holds all the selectors and actions for that page. The test files only describe the behaviour  they do not care about which button ID to click or how to find an input. This separation makes the suite easier to maintain. If demoblaze updates a selector, I change it in one place and every test that uses that page automatically picks up the fix.

For the purchase tests I log in through the API instead of the UI. The login UI is already covered in full inside `login.cy.js`, so driving through it again in every purchase test would just add time without testing anything new. The API call sets the session token in localStorage and the app treats it the same as a normal login.

Product selection is done dynamically where possible — tests read whatever product name is currently showing first on the page instead of hardcoding a specific product name. This means those tests keep working even if demoblaze changes the order products appear in.

For the cart total verification I added a `getCartTotal()` method in `CartPage.js` that reads the total directly from the page and returns it. Before placing an order, the test calls this method and passes the value through to `verifyOrderConfirmation()`. This means the confirmation popup is checked against the actual cart total rather than a hardcoded number, so if the price of a product changes the test still works correctly.

Category navigation uses `cy.intercept('POST', '**/bycat')` to wait for the API response before asserting product cards. This prevents the test from reading product cards from the previous category before the new ones have loaded.

### Things I found worth noting

While exploring the site before writing the tests, I noticed that the purchase form only validates two fields  Name and Credit card. Everything else is optional and gets ignored if left blank. More interestingly, there is no validation on the credit card number itself any combination of characters is accepted and the order goes through.

I also noticed that the order confirmation popup sometimes shows "Amount: 0 USD" instead of the actual purchase amount. This appears to be an inconsistency on demoblaze's backend rather than a bug in the tests. I have intentionally kept that assertion loose just checking that the Amount field exists so the tests do not fail randomly because of something outside our control.

The purchase tests use API login via a custom `cy.loginViaApi()` command rather than repeating the UI login flow in every test. While the API login works correctly and all tests pass, the Cypress preview window does not visually show the logged in state (Welcome username) during the test run. This is because demoblaze uses browser cookies to display the logged in UI state, and these are server generated session values that cannot be replicated outside of a full UI login. The API login still sets the correct token for all test operations to work the missing welcome message in the preview is a cosmetic limitation of the approach, not a functional one.

## AI Tool Usage Disclosure

**Alert stubbing behaviour** — During test execution, browser alert assertions using `cy.on('window:alert')` were producing unexpected behaviour tests were passing even when the assertion inside the alert callback was wrong. After running into this issue I looked it up with AI assistance, which explained that `cy.on('window:alert')` doesn't properly integrate with Cypress's command queue and can silently swallow failed assertions. The fix was to use `cy.stub()` instead, which hooks into Cypress properly and makes sure a failed assertion actually fails the test.

**Double stubbing issue on the product page** When asserting the "Product added." alert on the product page, there was a conflict between the alert stub set up at the start of the test (for the signup alert) and the stub needed later for the add-to-cart alert. Both were trying to use the same `window.alert` on what had become different window objects across page navigations. AI assistance was used to understand why the stub was stale and to find the right fix setting up a fresh `cy.window()` stub with a separate alias (`@addToCartAlert`) immediately before clicking Add to cart, and pairing it with `cy.intercept('POST', '**/addtocart')` to wait for the network call before asserting the alert. This resolved both the stale stub and the race condition in one go.

Everything else the test design, test cases, page structure, selectors, and debugging is my own work.
